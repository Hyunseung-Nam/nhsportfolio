# 포트폴리오 리디자인 노트 (prompt.txt 준수)

## 1) 기존 포트폴리오 문제점 요약
- 첫 화면에 정보가 분산되어 "유료 납품 2건"이 즉시 드러나는 강도가 부족했음.
- 프로젝트 설명 구조가 카드마다 달라 면접관이 빠르게 비교하기 어려웠음.
- 납품 프로젝트와 개인 프로젝트의 시각적 위계가 충분히 분리되지 않았음.
- 기술 스택 설명이 "어디에 썼는가"보다 "무엇을 쓸 줄 아는가"에 가까웠음.
- 협업/업무 스타일이 별도 섹션으로 명확히 정리되지 않아 실무 신뢰 판단 포인트가 약했음.

## 2) 수정 후 추천 구조
1. Hero (정체성 1문장 + 납품 2건 명시 + 버튼 2개)
2. 실사용 납품 프로젝트 (코인 세탁소 CRM, PC 사양 확인 프로그램)
3. 문제 해결 방식 / 강점
4. 개인 프로젝트 (실사용 전제 설계)
5. 기술 스택 (활용 맥락 중심)
6. 협업 / 업무 스타일
7. 연락 섹션

## 3) 각 섹션 수정 방향 및 문구 예시
### Hero
- 방향: 소개 문단/기술 나열을 줄이고 실무 판단 핵심만 유지
- 예시: "실사용 환경에서 문제를 해결하고, 납품 후 운영까지 책임지는 백엔드 개발자"

### 실사용 납품 프로젝트
- 방향: 2개 프로젝트를 최상단 배치하고 공통 템플릿(요약/문제/해결/결과/기술)으로 통일
- 예시(코인 세탁소 CRM)
  - 요약: "수기 포인트 관리 오류를 줄이기 위해 매장 운영형 CRM 구축"
  - 문제: "계산 실수, 누락, 기준 불일치"
  - 해결: "입력 흐름 단순화 + 기준 로직 고정"
  - 결과: "실사용 + 사용자 피드백 기반 업데이트"

### 문제 해결 방식 / 강점
- 방향: 성향 표현 제거, 운영 관점의 행동 문장으로 치환
- 예시: "요청을 기능 단위로 분해하고 영향 범위를 먼저 정리한 뒤 구현"

### 개인 프로젝트
- 방향: 연습/학습 표현 제거, 실사용 전제 설계로 정의
- 예시: "반복 업무 시간을 줄이기 위한 자동화 흐름을 재사용 가능한 구조로 구현"

### 기술 스택
- 방향: 기술명 나열 대신 프로젝트 맥락 연결
- 예시: "Python: PC 사양 조회와 자동화 로직에서 예외 처리와 실행 흐름 제어에 사용"

### 협업 / 업무 스타일
- 방향: 요구사항 정리, 커뮤니케이션 기준, 유지보수 인식을 항목화
- 예시: "변경 요청 시 영향 범위·일정·대안 3가지를 함께 공유"

### 연락 섹션
- 방향: 수단 1~2개만 유지, 마무리 문장으로 실사용 문제 해결 관점 강조
- 예시: "실사용 환경에서 발생하는 문제를 구조적으로 해결하는 개발 업무에 집중"

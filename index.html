실사용 환경에서 “안정적으로 동작하는 프로그램”을 만드는 개발자

유료 납품 프로젝트 2건(코인 세탁소 CRM, PC 사양 확인 프로그램)을 실제 운영 환경에 적용했고, 운영 중 요청·이슈를 정리해 업데이트로 반영했습니다.

보여주기용 기능보다, 운영에서 반복되는 실수/누락을 줄이는 설계와 환경 차이에 강한 수집·예외 처리를 우선합니다.

실사용 납품 프로젝트

아래 2개 프로젝트는 납품 후 실제 현장에서 운영된 사례입니다. (고객/매장 정보는 비공개 원칙을 따릅니다.)

1) 코인 세탁소 고객 관리 프로그램 (CRM)

Tags: Delivered · In Use · Maintained

한 줄 요약: 수기 포인트 관리에서 발생하던 오류·누락을 줄이기 위해 매장 운영형 CRM을 구축했습니다.

문제: 수기 관리로 계산 실수, 누락, 기준 불일치가 발생해 운영 부담이 커졌습니다.

해결: 운영자가 최소 입력으로 처리하도록 흐름을 단순화하고, 데이터 처리 기준을 로직으로 고정했습니다.

결과: 실제 매장에서 사용 중이며, 운영 흐름에서 드러난 UX 이슈를 정리해 업데이트를 지속 적용했습니다.

현장 피드백 → UX 개선 반영 (요약)

  • 검색 UX: 전화번호 입력 후 엔터로 검색되길 원함 → 엔터/클릭 검색 모두 지원
  • 가독성: 전화번호 숫자 구분 어려움 → 자동 하이픈 포맷 적용(입력/리스트 동일 기준)
  • 시인성: 주요 숫자 확인이 번거로움 → 크기·가중치 조정

※ 피드백은 원문 캡처가 아닌, 개선 항목 중심으로 정리했습니다.

기술: Python(UI/로직), 데이터 처리 로직(기준 고정), 로컬 배포/업데이트 관리, 사용자 피드백 기반 UX 개선

2) PC 사양 확인 프로그램

Tags: Delivered · Signed · Maintained

한 줄 요약: 환경별 조회 실패를 줄이기 위해 다중 조회 + fallback 구조를 적용한 사양 확인 도구를 개발했습니다.

문제: PC 환경별 차이로 단일 방식 조회만으로는 시스템 정보(CPU/GPU/RAM 등) 수집 안정성이 떨어질 수 있었습니다.

해결: 여러 수집 경로를 조합하고, 실패 조건을 기준으로 대체 경로로 자동 전환되도록 설계했습니다. 예외 처리/로그를 포함해 유지보수 관점으로 구조화했습니다.

결과: 다양한 환경에서 안정적으로 동작하는 형태로 배포했으며, 배포 신뢰성 강화를 위해 코드 서명을 적용했습니다. 보호 강화를 위해 난독화/워터마킹을 단계적으로 적용 중입니다.

기술: Python, 시스템 정보 수집 로직, 다중 조회·fallback 설계, 배포 환경 고려(코드 서명), 유지보수 대응(예외 처리/로그)

문제 해결 방식 / 강점

비개발자 대상 UX

입력 단순화

현장 사용자가 “설명 없이도” 흐름을 따라갈 수 있도록 입력 단계·용어를 줄이고 기준을 고정합니다.

운영 안정성

실패 시나리오

환경 차이·예외 케이스를 전제로 설계하고, 실패 조건에 따라 대체 경로로 전환되도록 구성합니다.

유지보수 관점

재현 → 수정

이슈는 먼저 재현 조건을 좁히고, 수정 범위와 배포 단위를 정해 업데이트로 반영합니다.

개인 프로젝트 (업무 흐름 개선 중심)

아래 항목은 납품 프로젝트와 구분되는 개인 프로젝트입니다. “보고 누락/수기 정리” 같은 반복 문제를 줄이기 위한 흐름 설계에 집중했습니다.

소규모 ERP (로그인 연동)

한 줄 요약: 보고 누락과 수기 정리로 인한 혼선을 줄이기 위해 로그인 기반 소규모 ERP를 설계·구현했습니다.

문제: 업무 보고가 여러 채널로 흩어져 누락/중복이 발생했고, 정리 작업이 반복되었습니다.

해결: 로그인 기반 사용자 구분 구조를 구성하고, 입력·저장·조회 흐름을 일관되게 만들어 보고 상태를 한 곳에서 관리하도록 했습니다.

결과: 보고 상태를 한 화면에서 확인할 수 있도록 하여, 누락 가능성을 낮추는 구조를 만들었습니다.

기술: Python, 인증/로그인 로직, CRUD 흐름 설계, 데이터 관리 구조, 화면 흐름 구성

기술 스택 (활용 맥락)

Python

납품 프로젝트와 ERP에서 UI/로직 구현, 예외 처리, 데이터 처리 기준 고정, 운영 피드백 반영에 사용했습니다.

시스템/환경 대응

환경 차이를 전제로 다중 조회·fallback을 설계하고, 로그로 원인 추적이 가능하도록 구성했습니다.

배포 / 운영

로컬 배포 환경 구성, 업데이트 반영, 운영 중 이슈 재현 및 수정 배포 흐름을 관리했습니다.

코드 서명

구조화 / 유지보수

모듈을 나누고 실패 조건을 명확히 해, 수정 범위를 좁히기 쉬운 형태로 구성했습니다.

업무 스타일

  • 요청을 받을 때: “무엇을 만들까”보다 “왜 필요한가 / 어디서 문제가 나는가”를 먼저 정리합니다.
  • 변경이 생길 때: 영향 범위를 확인하고, 선택지(그대로/대안/추가 작업)를 함께 공유합니다.
  • 범위를 잡을 때: 초기 구현 범위와 이후 개선 범위를 구분해 운영 중 혼선을 줄입니다.
  • 이슈 대응: 재현 조건을 좁힌 뒤, 수정 범위와 배포 단위를 결정합니다.
